**Date:** 2026-05-13
**Status:** Draft (v0.2 collapses v0.2/v0.3/v0.4-without-Cloud-NAT into a single release)
**Owner:** Nishant Gupta
**Builds on:** [`2026-05-13-herm-design.md`](2026-05-13-herm-design.md) (the v0.1 design)

## 1. What v0.2 adds on top of v0.1

v0.1 shipped a VM running Hermes Agent v0.13.0 reachable on the tailnet at `:8642` with bearer auth. v0.2 adds:

1. **Multica orchestration layer.** Multica runs on the VM and registers Hermes as its primary agent. The user-facing surface becomes Multica (tickets + heartbeats + skills + budget) rather than Hermes directly. Hermes remains the brain; Multica is the foreman.
2. **Skills system.** YAML skill definitions in the repo (under `skills/`) seeded into Multica on first boot. Initial set: `debug`, `review-pr`, `write-doc`, `update-deps`, `watch-repo`, `summarize-day`. Each skill bundles system prompt + tool allowlist + memory scope + optional schedule + budget cap.
3. **Full `herm` CLI.** `login {claude,gh,gemini,codex,opencode,goose,all}`, `open`, `qr`, `rotate`, `upgrade`, `backup`, `restore`, `logs`, `console` — plus the v0.1 commands carried forward.
4. **Hardening (non-Cloud-NAT subset of the original v0.4 paranoid mode).**
   - Tool allowlist on Hermes' gateway (no blanket shell).
   - Auto-reaper: VM auto-deletes after N hours of no tailnet activity from owner devices.
   - `herm rotate` regenerates the Hermes API server token and Multica's bearer (separate tokens).
   - Optional CMEK on disk / GCS bucket / Secret Manager via `herm up --cmek`.
   - Disk snapshot before `herm upgrade`, automatic rollback if the upgraded units fail to start.

## 2. Explicit non-goals for v0.2

- **Cloud NAT.** Owner rejected ~$32/mo cost. Egress-allowlist paranoid mode deferred.
- **Per-agent Podman sandboxing.** Significant complexity (rootless Podman config, per-agent UIDs, cross-container token sharing). Deferred.
- **AionUI / Hermes Desktop wiring on the laptop side.** v0.2 makes the Multica web UI the primary surface — desktop native clients are a v0.3 polish task.
- **Mobile push notifications.** Multica's heartbeat events stay in-VM; pushing to phone is later.
- **Multi-cloud abstraction.** Still GCP only.

## 3. Architecture (delta from v0.1)

```
You ──► Multica web UI :3000 ──► tickets, heartbeats, skill registry, budget tracking
                                       │
                                       └─► Hermes Agent :8642  ──► persistent memory, reasoning loop
                                                                │
                                                                └─► claude / gh / gemini / codex / opencode / goose
                                                                    (subprocess delegation when a skill says so)

Auto-reaper systemd timer ──► tailscale status check ──► if no owner devices seen for N hours, halt + cleanup
```

Multica runs as its own systemd unit (`multica.service`) under the same `herm` user. Both gateways listen on `0.0.0.0` and are exposed only via the tailnet. Multica calls Hermes' `:8642/v1/chat/completions` with the per-Hermes bearer token internally.

## 4. Skills system

### 4.1 Skill YAML schema

```yaml
# skills/<name>.yaml
name: review-pr
description: One-sentence summary of what this skill does.

# How the agent reasons inside the skill:
system_prompt: |
  You are a focused PR reviewer. Read the diff, the linked issue if any,
  and the changed-file context. Output a single review comment...

# Which tools the agent may use inside the skill. Empty list = no tools (text-only).
tools:
  - gh:read
  - claude:code-review
  - filesystem:read

# Optional: pin a specific model for this skill (e.g., bigger context for refactors).
model: claude-sonnet-4-6

# Optional: budget ceiling per invocation in USD.
budget_usd: 0.10

# Optional: cron schedule for heartbeat skills.
schedule: "*/30 * * * *"

# Optional: scope for memory writes (per-skill, per-project, or "global").
memory_scope: per-skill
```

### 4.2 Initial skill set (in `skills/` in the repo)

| File | Purpose |
|---|---|
| `debug.yaml` | Given an error + repro, narrow down the cause and propose a minimal fix. Uses Claude Code + read/write to active repo. |
| `review-pr.yaml` | Read a PR diff, post a structured comment via `gh`. Read-only on the repo. |
| `write-doc.yaml` | Generate or edit a doc file under `docs/`. No shell, only file write under `docs/`. |
| `update-deps.yaml` | Open a PR upgrading deps in a worktree (does not commit to main). |
| `watch-repo.yaml` | `gh` polling every 30 min. Drops a note in the agent's inbox when interesting events land. |
| `summarize-day.yaml` | Cron at `09:00`. Reads yesterday's memory + `gh` notifications, writes a brief into the inbox. |

### 4.3 Skill loader

On boot, `99-systemd-units.sh` rsyncs `/opt/herm/skills/*.yaml` to `/home/herm/.multica/skills/`. Multica reads from there. Future skill additions on the VM (via `multica skill add` or by dropping a YAML in that directory) are preserved across `herm down/up` because they live on the persistent disk.

## 5. CLI expansion

### 5.1 New subcommands

```
herm login claude       SSH in, run `claude setup-token`, persist on disk
herm login gh           SSH in, run `gh auth login --web` (device-code)
herm login gemini       SSH in, run `gemini auth login`
herm login codex        SSH in, run `codex login`
herm login opencode     SSH in, run `opencode login`
herm login goose        SSH in, run `goose login`
herm login all          Run the above sequentially, with a clear failure summary at the end

herm open multica       Open http://herm-vm:3000 in default browser
herm open hermes        Open http://herm-vm:8642 (returns JSON / API docs)
herm open ttyd          (Deferred to v0.3 — ttyd is not yet shipped; placeholder)

herm qr                 Print a terminal QR code with the Multica URL + API key — scan from phone

herm rotate             Generate fresh tokens for Hermes API and Multica, restart units, print new values
herm upgrade            Snapshot disk → pull pinned-version updates → restart units → verify; auto-rollback on failure
herm backup now         Trigger the GCS rsync immediately (bypass the timer)
herm restore <ts>       Restore /home/herm from a backup snapshot identified by ISO-8601 timestamp
herm logs <unit>        `gcloud compute ssh ... -- sudo journalctl -u <unit> -f` (Hermes, Multica, herm-backup)
herm console            Opens the GCP console for the VM in default browser
```

### 5.2 Argument parsing

Each subcommand parses its own flags inside `cli/commands/<cmd>.sh`. `bin/herm` only routes by name. Flags use long-form (e.g., `--paranoid`, `--cmek`, `--force`) — no short options in v0.2.

## 6. Hardening (non-Cloud-NAT subset)

### 6.1 Tool allowlist for Hermes' gateway

Configure Hermes' tool registry to enable only:

- `web_search` (read-only, low risk)
- `code_execution` (sandboxed Python via Hermes' built-in)
- `delegation` (skill subagents)
- `clarify` (ask the user)
- `cronjob` (schedule heartbeats)
- `memory` (read/write own memory store)
- `session_search`
- `task_planning`
- `mcp:native-mcp` (allow MCP tools)
- `github` (limited via skill allowlist)

Disable:
- `computer_use` (could click anything)
- `browser_cdp` (full browser control)
- `discord`, `slack`, `email`, `xurl` (deferred until external messaging is wired)
- `godmode` (red-teaming skill — explicit opt-in only)

Implementation: edit `/home/herm/.hermes/config.yaml` `tools.enabled` list during install. Configurable via `config/hermes-tools.yaml` in the repo so users can adjust the policy in their fork.

### 6.2 Auto-reaper

A `herm-reaper.timer` runs every 30 min. The `herm-reaper.service` calls `tailscale status --json`, counts owner-tagged active peers in the last N hours (default 168 = 7 days), and if zero, calls `systemctl poweroff`. The VM's instance schedule is configured to auto-restart on owner reconnect — actually no: GCE doesn't have "auto-start on tailscale reconnect." Instead, `herm up` is the manual restart. Document this clearly.

Configurable via `~/.config/herm/config.toml`:

```toml
[reaper]
enabled = false                 # default off
idle_hours = 168                # 7 days
```

### 6.3 `herm rotate`

```
herm rotate                Rotate the Hermes API server token AND the Multica admin token
herm rotate hermes         Rotate only the Hermes API server token
herm rotate multica        Rotate only the Multica admin token
```

For each rotation:
1. Generate a new 32-byte random base64 token on the VM.
2. Write to the respective `.env` file.
3. Restart the systemd unit via IAP-SSH + sudo.
4. Verify the new token works against `/health`.
5. Print the new token in the laptop terminal, suffixed with "**copy this to Hermes Desktop / Multica clients now; the value is not re-displayable**."

### 6.4 Optional CMEK

`herm up --cmek` adds:
- A KMS keyring + key in the same region.
- Customer-managed encryption on the PD-SSD, GCS bucket, Secret Manager entries.
- IAM bindings so the VM service account has `cloudkms.cryptoKeyDecrypter` on that key only.

Toggle is sticky: once a project is provisioned with `--cmek`, subsequent `herm up` cycles default to using the same key. Reverting requires `herm nuke && herm up`.

### 6.5 `herm upgrade` with disk snapshot

1. Snapshot the PD-SSD to GCE Snapshots (taken via Terraform `google_compute_snapshot` or via `gcloud compute snapshots create` from the CLI command — pick whichever is more idempotent).
2. Pull latest pinned versions of Hermes / Multica / per-skill deps.
3. Restart units.
4. Curl Hermes `/health` and Multica `/health`. If either fails for 5 consecutive checks across 30 seconds, automatic rollback: stop the units, restore the previous snapshot, restart.
5. On success, prune snapshots older than 7 days to bound storage cost.

## 8. Success criteria for v0.2

1. **End-to-end Multica → Hermes path.** A request to Multica's API on `:3000/v1/chat/completions` (or wherever Multica exposes OpenAI-compat) routes to Hermes' `:8642` internally, gets an answer from Claude, returns to the caller.
2. **Skill invocation.** Sending `"!debug <stack trace>"` (or whatever Multica's skill invocation syntax is) routes through Multica → Hermes-with-debug-skill → real model response.
3. **`herm rotate` works** without downing Hermes or Multica for more than 5s each.
4. **Auto-reaper does not fire** while the owner's laptop is active on the tailnet; fires after explicit `tailscale logout` on all owner devices > idle_hours.
5. **`herm upgrade` rolls back** automatically when a deliberately-corrupted upgrade fails to start.
6. **CI green** on all new code: shellcheck, terraform validate/tflint, bats.
7. **Fresh-clone integration test** (could be done by tearing down + re-upping `flashckard`): new clone → `herm init` → `herm up` → Multica reachable → skill executes. v0.2 is functionally complete when this round-trip works end-to-end on a clean VM.

## 9. Decision log

- **Multi-agent identity stays A2** (single visible `nishant-agent`, invisible specialist subagents via skills). Same as v0.1 spec.
- **Multica vs. Paperclip** stays Multica. Paperclip is too org-chart-shaped for "me as one agent."
- **Cloud NAT** explicitly deferred. The cost ($32/mo) made the owner balk; v0.2 ships without true egress-allowlist paranoid mode.
- **Multica on the same VM** rather than a separate VM. Both run as `herm`; one machine, one tailnet entry, no inter-VM auth complexity.
- **Tool allowlist as v0.2 default** rather than opt-in. Earlier the spec called this paranoid-mode-only; pulling it into the default because a tool-use Hermes with `computer_use` enabled on a credential-holding VM is more risk than a Linux-permission deny-all-default warrants.

## 10. Migration from v0.1

A user running v0.1.0 today can take v0.2.0 with:

```
git pull && git checkout v0.2.0
herm upgrade
```

`herm upgrade` snapshots the disk, runs the v0.2 install scripts on the existing VM (Multica + skill seeding + tool-allowlist patch + reaper timer install + token-rotation primitive), and restarts the units. Existing Hermes memory, OAuth tokens, and skill files survive because they all live on the persistent disk. Rollback path: the snapshot from before the upgrade is named `herm-pre-v0.2-<timestamp>`.
