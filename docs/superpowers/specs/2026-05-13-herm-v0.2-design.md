**Date:** 2026-05-13
**Status:** Draft. Pivot from earlier draft: Multica deferred to v0.3 after live research uncovered architectural mismatch with the original v0.1 brainstorm's assumption.
**Owner:** Nishant Gupta

## 0. Why this spec is different from the brainstorm

The v0.1 brainstorm picked Multica as the orchestration layer on the assumption that Multica registers Hermes via an HTTP+Bearer interface and routes user requests through it. Live research against the upstream docs found:

- Multica is a **task/issue board over WebSocket** with a local daemon that **spawns CLIs as subprocesses on the daemon host** ‚Äî not an HTTP router.
- Multica skills use **Anthropic Agent-Skills format** (`SKILL.md` Markdown with frontmatter), the same format Hermes Agent v0.13.0 already consumes.
- There is no documented `Authorization: Bearer` outbound HTTP integration from Multica to a registered agent. `/docs/agents/http` and `/docs/runtimes` on multica.ai both 404.

Hermes Agent v0.13.0 natively provides the four properties the brainstorm wanted from Multica:

| Brainstorm requirement | Hermes-native primitive |
|---|---|
| Tickets / a queue | Hermes sessions + `task_planning` toolset |
| Heartbeats / schedules | Hermes `cronjob` toolset |
| Skills that compound | `~/.hermes/skills/` (Anthropic Agent-Skills format) ‚Äî already 82 bundled with v0.13.0 |
| Specialist subagents (A2 invisible-delegation model) | Hermes `delegation` toolset |

v0.2 therefore ships **Hermes-native skills + scheduling**, the full `herm` CLI surface, and the non-Cloud-NAT hardening ‚Äî *without* a separate Multica process. Multica returns in v0.3 once its actual integration shape has been verified by reading `multica-ai/multica` source (not the marketing docs).

## 1. What v0.2 adds on top of v0.1

1. **Custom skills** dropped into `~/.hermes/skills/` on the VM, seeded from `skills/` in the repo on first boot. Initial set: `debug`, `review-pr`, `write-doc`, `update-deps`, `watch-repo`, `summarize-day`. Each skill is an Anthropic Agent-Skills `SKILL.md` with frontmatter.
2. **Heartbeat / scheduled skills** registered via Hermes' `cronjob` toolset ‚Äî driven by `schedule:` frontmatter in the SKILL.md.
3. **Full `herm` CLI.** `login {claude,gh,gemini,codex,opencode,goose,all}`, `open`, `qr`, `rotate`, `upgrade`, `backup`, `restore`, `logs`, `console` ‚Äî plus the v0.1 commands carried forward.
4. **Hardening (non-Cloud-NAT subset of the original v0.4 paranoid mode).**
   - Tool allowlist on Hermes' gateway (no blanket shell / `computer_use` / `godmode`).
   - Auto-reaper: VM auto-halts after N hours of no tailnet activity from owner devices.
   - `herm rotate` regenerates the Hermes API server token.
   - Optional CMEK on disk / GCS bucket / Secret Manager via `herm up --cmek`.
   - Disk snapshot before `herm upgrade`, automatic rollback if the upgraded units fail to start.

## 2. Explicit non-goals for v0.2

- **Cloud NAT.** Owner rejected ~$32/mo cost. Egress-allowlist paranoid mode deferred.
- **Per-agent Podman sandboxing.** Significant complexity (rootless Podman config, per-agent UIDs, cross-container token sharing). Deferred.
- **Multica.** Architectural shape doesn't match the brainstorm's assumption (see ¬ß0). Returning in v0.3 after source-reading.
- **AionUI / Hermes Desktop wiring on the laptop side.** v0.2 keeps the Hermes TUI + HTTP API as the surface; desktop native clients are a v0.3 polish task.
- **Mobile push notifications.** Heartbeat events stay in-VM; pushing to phone is later.
- **Multi-cloud abstraction.** Still GCP only.

No new long-running daemons in v0.2 ‚Äî Hermes is still the only agent process. v0.2 adds new *skills* that Hermes loads at startup, plus three small systemd units (the reaper timer/service, and a rotated `hermes-agent.service` after `herm rotate` writes a new token).

## 4. Skills system

Hermes consumes **Anthropic Agent-Skills** format: a directory with a `SKILL.md` (Markdown + YAML frontmatter) plus optional supporting files. v0.13.0 already bundles 82 skills under `~/.hermes/skills/` ‚Äî we add a `herm/` subdirectory for the skills we ship.

### 4.1 SKILL.md schema (Anthropic format, what Hermes actually consumes)

```markdown
---
name: review-pr
description: Review a GitHub PR by reading the diff, the linked issue, and the changed-file context, then post a structured comment.
tools: [gh, read_file, web_fetch]
model: claude-sonnet-4-6
schedule: ""             # not a heartbeat skill
budget_usd: 0.10
---

# review-pr

When the user asks "review PR X" or pastes a PR URL, do this:

1. `gh pr view <url>` to read the description.
2. `gh pr diff <url>` to read the diff.
3. ...

## Style

- One coherent comment, not line-by-line nitpicks unless a real bug.
- Lead with the highest-impact concern.
- Cite specific file:line locations.
```

The frontmatter `tools:` whitelist intersects with Hermes' global tool allowlist (¬ß6.1); intersection is what the skill can use. `schedule:` non-empty registers the skill with Hermes' `cronjob` toolset on startup.

### 4.2 Initial skill set (in `skills/` in the repo)

| Dir | Purpose |
|---|---|
| `skills/debug/SKILL.md` | Given an error + repro, narrow down the cause and propose a minimal fix. Uses Claude Code via subprocess delegation. |
| `skills/review-pr/SKILL.md` | Read a PR diff, post a structured comment via `gh`. |
| `skills/write-doc/SKILL.md` | Generate or edit a doc file. File-write only under `docs/`. No shell. |
| `skills/update-deps/SKILL.md` | Open a PR upgrading deps in a worktree. Doesn't commit to main. |
| `skills/watch-repo/SKILL.md` | `gh` polling every 30 min via cron. Notes to inbox on interesting events. |
| `skills/summarize-day/SKILL.md` | Cron at `09:00 UTC`. Reads yesterday's memory + `gh` notifications. |

### 4.3 Skill loader

The startup script's `07-seed-skills.sh` rsyncs `/opt/herm/skills/` ‚Üí `/home/herm/.hermes/skills/herm/`. Hermes auto-discovers skills under `~/.hermes/skills/` on next start. Future user-authored skills can live in the same path; `herm up` does not overwrite them (the rsync uses `--ignore-existing` for the user-owned scope and only blasts `herm/` subdirectory).

## 5. CLI expansion

### 5.1 New subcommands

```
herm login claude       SSH in, run `claude setup-token`, persist on disk
herm login gh           SSH in, run `gh auth login --web` (device-code)
herm login gemini       SSH in, run `gemini auth login`
herm login codex        SSH in, run `codex login`
herm login opencode     SSH in, run `opencode login`
herm login goose        SSH in, run `goose login`
herm login all          Run the above sequentially, with a clear failure summary at the end

herm open multica       Open http://herm-vm:3000 in default browser
herm open hermes        Open http://herm-vm:8642 (returns JSON / API docs)
herm open ttyd          (Deferred to v0.3 ‚Äî ttyd is not yet shipped; placeholder)

herm qr                 Print a terminal QR code with the Multica URL + API key ‚Äî scan from phone

herm rotate             Generate fresh tokens for Hermes API and Multica, restart units, print new values
herm upgrade            Snapshot disk ‚Üí pull pinned-version updates ‚Üí restart units ‚Üí verify; auto-rollback on failure
herm backup now         Trigger the GCS rsync immediately (bypass the timer)
herm restore <ts>       Restore /home/herm from a backup snapshot identified by ISO-8601 timestamp
herm logs <unit>        `gcloud compute ssh ... -- sudo journalctl -u <unit> -f` (Hermes, Multica, herm-backup)
herm console            Opens the GCP console for the VM in default browser
```

### 5.2 Argument parsing

Each subcommand parses its own flags inside `cli/commands/<cmd>.sh`. `bin/herm` only routes by name. Flags use long-form (e.g., `--paranoid`, `--cmek`, `--force`) ‚Äî no short options in v0.2.

## 6. Hardening (non-Cloud-NAT subset)

### 6.1 Tool allowlist for Hermes' gateway

Configure Hermes' tool registry to enable only:

- `web_search` (read-only, low risk)
- `code_execution` (sandboxed Python via Hermes' built-in)
- `delegation` (skill subagents)
- `clarify` (ask the user)
- `cronjob` (schedule heartbeats)
- `memory` (read/write own memory store)
- `session_search`
- `task_planning`
- `mcp:native-mcp` (allow MCP tools)
- `github` (limited via skill allowlist)

Disable:

- `computer_use` (could click anything)
- `browser_cdp` (full browser control)
- `discord`, `slack`, `email`, `xurl` (deferred until external messaging is wired)
- `godmode` (red-teaming skill ‚Äî explicit opt-in only)

Implementation: edit `/home/herm/.hermes/config.yaml` `tools.enabled` list during install. Configurable via `config/hermes-tools.yaml` in the repo so users can adjust the policy in their fork.

### 6.2 Auto-reaper

A `herm-reaper.timer` runs every 30 min. The `herm-reaper.service` calls `tailscale status --json`, counts owner-tagged active peers in the last N hours (default 168 = 7 days), and if zero, calls `systemctl poweroff`. The VM's instance schedule is configured to auto-restart on owner reconnect ‚Äî actually no: GCE doesn't have "auto-start on tailscale reconnect." Instead, `herm up` is the manual restart. Document this clearly.

Configurable via `~/.config/herm/config.toml`:

```toml
[reaper]
enabled = false                 # default off
idle_hours = 168                # 7 days
```

### 6.3 `herm rotate`

```
herm rotate                Rotate the Hermes API server token
```

(Multica subcommand parked for v0.3.)

For each rotation:

1. Generate a new 32-byte random base64 token on the VM.
2. Write to the respective `.env` file.
3. Restart the systemd unit via IAP-SSH + sudo.
4. Verify the new token works against `/health`.
5. Print the new token in the laptop terminal, suffixed with "**copy this to Hermes Desktop / Multica clients now; the value is not re-displayable**."

### 6.4 Optional CMEK

`herm up --cmek` adds:

- A KMS keyring + key in the same region.
- Customer-managed encryption on the PD-SSD, GCS bucket, Secret Manager entries.
- IAM bindings so the VM service account has `cloudkms.cryptoKeyDecrypter` on that key only.

Toggle is sticky: once a project is provisioned with `--cmek`, subsequent `herm up` cycles default to using the same key. Reverting requires `herm nuke && herm up`.

### 6.5 `herm upgrade` with disk snapshot

1. Snapshot the PD-SSD to GCE Snapshots (taken via Terraform `google_compute_snapshot` or via `gcloud compute snapshots create` from the CLI command ‚Äî pick whichever is more idempotent).
2. Pull latest pinned versions of Hermes / Multica / per-skill deps.
3. Restart units.
4. Curl Hermes `/health` and Multica `/health`. If either fails for 5 consecutive checks across 30 seconds, automatic rollback: stop the units, restore the previous snapshot, restart.
5. On success, prune snapshots older than 7 days to bound storage cost.

## 8. Success criteria for v0.2

1. **Skill discovery.** `hermes` (the TUI) shows our 6 new skills under "Available Skills" alongside the bundled ones.
2. **Skill invocation.** A test prompt that maps to one of our skills (e.g., "summarize today's GitHub activity") gets routed through the skill and produces real output backed by `gh`.
3. **`herm rotate` works** without downing Hermes for more than 5s; the new token successfully authenticates against `/health` and the old one is rejected within 1 minute.
4. **Auto-reaper does not fire** while the owner's laptop is active on the tailnet; **does fire** when a synthetic "no peers for idle_hours" condition is met.
5. **`herm upgrade` rolls back** automatically when a deliberately-corrupted upgrade fails to start.
6. **CI green** on all new code: shellcheck, terraform validate/tflint, bats.
7. **Fresh-clone integration test** (via `herm up --replace-vm` on `flashckard`): rebuild VM ‚Üí skills auto-seed ‚Üí tool allowlist applied ‚Üí end-to-end chat with Sonnet 4.6 still works. v0.2 is shippable when this passes.

## 9. Decision log

- **Multi-agent identity stays A2** (single visible `nishant-agent`, invisible specialist subagents via skills). Same as v0.1 spec.
- **Multica deferred to v0.3.** Live research (see ¬ß0) showed Multica is a task board with subprocess CLI dispatch, not the HTTP-bearer router the brainstorm assumed. Reintegration requires reading `multica-ai/multica` source to find the actual runtime registration path. Hermes' native primitives cover the v0.1 brainstorm requirements (tickets, schedules, skills, delegation) without it.
- **Paperclip stays out.** Too org-chart-shaped for "me as one agent."
- **Cloud NAT** explicitly deferred. The cost ($32/mo) made the owner balk; v0.2 ships without true egress-allowlist paranoid mode.
- **Tool allowlist as v0.2 default** rather than opt-in. A tool-use Hermes with `computer_use` enabled on a credential-holding VM is more risk than a Linux-permission deny-all warrants by itself.
- **Skill format = Anthropic Agent-Skills (Markdown + frontmatter)** rather than our own YAML. Hermes already consumes it; Multica would too if/when we add it.

## 10. Migration from v0.1

A user running v0.1.0 today can take v0.2.0 with:

```
git pull && git checkout v0.2.0
herm upgrade
```

`herm upgrade` snapshots the disk, runs the v0.2 install scripts on the existing VM (skill seeding + tool-allowlist patch + reaper timer install + token-rotation primitive), and restarts the units. Existing Hermes memory, OAuth tokens, and any user-authored skills survive because they all live on the persistent disk. Rollback path: the snapshot from before the upgrade is named `herm-pre-v0.2-<timestamp>`.
